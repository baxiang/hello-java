# 53.4 分布式锁：并发控制（生活类比：试衣间）

## 一、先问一个问题

**问题**：多用户同时下单，如何防止超卖？

在单机时代，我们可以用 `synchronized` 或 `ReentrantLock` 保证线程安全。但在分布式环境中，多个服务实例运行在不同的 JVM 上，本地锁失效了。

### 生活例子：试衣间

想象商场的试衣间：
- 一个人进去（获取锁）
- 门锁上，别人在外面等待（阻塞）
- 出来开门（释放锁）
- 下一个人进去

分布式锁就是多服务器的"试衣间门"！只有拿到钥匙的服务才能执行关键业务。

## 二、使用 Redisson

### 2.1 添加依赖

```xml
<dependency>
    <groupId>org.redisson</groupId>
    <artifactId>redisson-spring-boot-starter</artifactId>
    <version>3.21.3</version>
</dependency>
```

### 2.2 配置类

```java
@Configuration
public class RedissonConfig {
    
    @Bean
    public RedissonClient redissonClient() {
        Config config = new Config();
        config.useSingleServer()
              .setAddress("redis://localhost:6379")
              .setDatabase(0);
        return Redisson.create(config);
    }
}
```

### 2.3 使用分布式锁

```java
@Service
public class OrderService {
    
    @Autowired
    private RedissonClient redissonClient;
    
    public void createOrder(String userId) {
        RLock lock = redissonClient.getLock("order:lock:" + userId);
        
        try {
            // 尝试获取锁，最多等待 3 秒，锁定 10 秒后自动释放
            if (lock.tryLock(3, 10, TimeUnit.SECONDS)) {
                try {
                    // 业务逻辑：创建订单
                    System.out.println("处理订单，用户：" + userId);
                } finally {
                    lock.unlock();
                }
            } else {
                throw new RuntimeException("系统繁忙，请稍后重试");
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new RuntimeException("获取锁被中断");
        }
    }
}
```

Redisson 的核心优势：
- **可重入锁**：同一线程可多次获取
- **看门狗机制**：默认 30 秒自动续期，防止业务未执行完锁就过期
- **支持多种锁**：公平锁、读写锁、红锁等

## 三、使用 RedisTemplate 实现

如果不引入 Redisson，也可以用 RedisTemplate 实现简单的分布式锁。

### 3.1 锁工具类

```java
@Service
public class RedisLockService {
    
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;
    
    private static final String SCRIPT = 
        "if redis.call('get', KEYS[1]) == ARGV[1] then " +
        "    return redis.call('del', KEYS[1]) " +
        "else " +
        "    return 0 " +
        "end";
    
    // 获取锁
    public boolean tryLock(String key, String value, long expire) {
        Boolean success = redisTemplate.opsForValue()
                .setIfAbsent(key, value, expire, TimeUnit.SECONDS);
        return Boolean.TRUE.equals(success);
    }
    
    // 释放锁（Lua 脚本保证原子性）
    public boolean unlock(String key, String value) {
        Long result = redisTemplate.execute(
            new DefaultRedisScript<>(SCRIPT, Long.class),
            Collections.singletonList(key),
            value
        );
        return result != null && result == 1;
    }
}
```

### 3.2 使用示例

```java
@Service
public class InventoryService {
    
    @Autowired
    private RedisLockService lockService;
    
    public void decreaseStock(Long productId) {
        String lockKey = "stock:lock:" + productId;
        String lockValue = UUID.randomUUID().toString();
        
        if (lockService.tryLock(lockKey, lockValue, 30)) {
            try {
                // 扣减库存
                System.out.println("扣减库存，商品：" + productId);
            } finally {
                lockService.unlock(lockKey, lockValue);
            }
        } else {
            throw new RuntimeException("系统繁忙");
        }
    }
}
```

## 四、常见疑问

**Q1：Redisson 和 RedisTemplate 实现有什么区别？**
A：Redisson 功能更完善，支持可重入锁、看门狗自动续期、多种锁类型。RedisTemplate 实现简单，但需要自己处理续期和原子性问题。

**Q2：为什么释放锁要用 Lua 脚本比较 value？**
A：防止误删别人的锁。如果锁过期后业务还在执行，另一个线程可能已经获取了锁，此时直接删除会破坏并发安全。

**Q3：Redis 分布式锁的可靠性如何？**
A：单机 Redis 存在主从切换丢锁的风险。对可靠性要求极高的场景，可使用 Redlock 算法（Redisson 已实现）或改用 ZooKeeper/Etcd。

## 五、小结

| 实现方式 | 优点 | 缺点 | 适用场景 |
|----------|------|------|----------|
| Redisson | 功能完善、自动续期、支持多种锁 | 依赖第三方库 | 生产环境推荐 |
| RedisTemplate | 无需额外依赖、代码轻量 | 需自己处理续期和原子性 | 简单场景 |
