# 31.8 模块化系统：什么是 Jigsaw？

## 一、先问一个问题

**什么是 Java 模块化？**

### 生活例子：乐高积木

```
传统 Java（单体）
├── 整个 JAR 包
├── 所有类都可用
└── 臃肿

模块化 Java（乐高）
├── 按需选择模块
├── 只用需要的部分
└── 精简
```

**Java 9 引入模块化**：
- 更好的封装
- 更小的运行时
- 更清晰的依赖

## 二、模块定义

### module-info.java

```java
// 模块声明文件
module com.example.mymodule {
    // 依赖其他模块
    requires java.sql;
    requires java.logging;
    
    // 导出包（其他模块可以访问）
    exports com.example.api;
    exports com.example.service;
    
    // 开放包（允许反射）
    opens com.example.internal;
    
    // 使用服务
    uses com.example.spi.MyService;
    
    // 提供服务
    provides com.example.spi.MyService 
        with com.example.impl.MyServiceImpl;
}
```

### 模块指令

| 指令 | 说明 | 示例 |
|------|------|------|
| requires | 依赖模块 | requires java.sql; |
| exports | 导出包 | exports com.example.api; |
| opens | 开放包（反射） | opens com.example.internal; |
| uses | 使用服务 | uses com.example.spi.MyService; |
| provides | 提供服务 | provides ... with ...; |

## 三、模块类型

### 1. 系统模块

```java
// Java 平台模块
java.base         // 基础模块（所有模块都依赖）
java.sql          // SQL 模块
java.logging      // 日志模块
java.xml          // XML 模块
java.net.http     // HTTP 客户端模块
```

### 2. 应用模块

```java
// 自定义应用模块
module com.example.app {
    requires java.base;
    requires java.sql;
    
    exports com.example.app.api;
}
```

### 3. 自动模块

```java
// 传统 JAR 放入模块路径自动成为自动模块
// 模块名 = JAR 文件名
my-lib-1.0.jar → 自动模块 my.lib
```

### 4. 无名模块

```java
// 类路径上的所有 JAR 组成无名模块
// 可以访问所有导出的包
// 所有包都自动导出
```

## 四、模块编译和运行

### 编译模块

```bash
# 编译模块
javac -d out --module-source-path src \
    $(find src -name "*.java")

# 或编译特定模块
javac -d out/com.example.app \
    src/com.example.app/module-info.java \
    src/com.example.app/com/example/app/*.java
```

### 打包模块

```bash
# 创建 JAR
jar --file out/myapp.jar \
    --main-class com.example.app.Main \
    -C out/com.example.app .
```

### 运行模块

```bash
# 运行模块
java --module-path out \
    --module com.example.app/com.example.app.Main

# 或运行 JAR
java --module-path out \
    --jar out/myapp.jar
```

## 五、模块服务

### 服务接口

```java
// com.example.spi
package com.example.spi;

public interface MyService {
    void execute();
}
```

### 服务实现

```java
// com.example.impl
package com.example.impl;

import com.example.spi.MyService;

public class MyServiceImpl implements MyService {
    @Override
    public void execute() {
        System.out.println("执行服务");
    }
}
```

### 模块配置

```java
// 服务提供者模块
module com.example.impl {
    requires com.example.spi;
    
    provides com.example.spi.MyService 
        with com.example.impl.MyServiceImpl;
}

// 服务消费者模块
module com.example.app {
    requires com.example.spi;
    
    uses com.example.spi.MyService;
}
```

### 使用服务

```java
// 使用 ServiceLoader
ServiceLoader<MyService> loader = ServiceLoader.load(MyService.class);

for (MyService service : loader) {
    service.execute();
}

// 或 Java 9+ 简化写法
ServiceLoader.load(MyService.class).forEach(MyService::execute);
```

## 六、模块化迁移

### 逐步迁移策略

```
阶段 1：保持类路径兼容
├── 所有 JAR 在类路径
└── 作为无名模块运行

阶段 2：迁移到模块路径
├── 部分 JAR 移到模块路径
└── 作为自动模块运行

阶段 3：完全模块化
├── 添加 module-info.java
└── 显式声明依赖和导出
```

### 迁移命令

```bash
# 分析依赖
jdeps --analyze --multi-release 11 myapp.jar

# 生成 module-info
jdeps --generate-module-info out myapp.jar

# 查看模块图
jdeps --generate-open-module out myapp.jar
```

## 七、JLink 自定义运行时

### 创建精简运行时

```bash
# 创建包含所需模块的运行时
jlink --module-path out \
    --add-modules com.example.app \
    --output custom-runtime \
    --compress=2 \
    --no-header-files \
    --no-man-pages
```

### 使用自定义运行时

```bash
# 运行应用
./custom-runtime/bin/java \
    --module com.example.app/com.example.app.Main
```

### 优势

| 特性 | 完整 JRE | 自定义运行时 |
|------|----------|--------------|
| 大小 | ~200MB | ~30MB |
| 启动时间 | 慢 | 快 |
| 模块 | 全部 | 按需 |

## 八、小结

| 概念 | 说明 | 示例 |
|------|------|------|
| module-info | 模块描述符 | module com.example { } |
| requires | 模块依赖 | requires java.sql; |
| exports | 包导出 | exports com.example.api; |
| opens | 开放反射 | opens com.example.internal; |
| jlink | 自定义运行时 | jlink --add-modules |

**核心要点**：
- 更好的封装
- 更小的运行时
- 更清晰的依赖
- 更好的安全性

## 九、常见问题

### Q1：模块化有什么好处？

**好处**：
- 封装性：只暴露需要的 API
- 可靠性：编译时检查依赖
- 安全性：限制访问
- 可扩展性：更好的服务机制
- 性能：更小的运行时

### Q2：什么时候使用模块化？

**建议**：
- 新项目：直接使用模块化
- 大型项目：考虑模块化
- 小型项目：可以不用
- 库/框架：推荐模块化

### Q3：如何处理反射？

```java
// 方式 1：开放包
module com.example {
    opens com.example.internal;
}

// 方式 2：运行时添加
java --add-opens com.example/com.example.internal=ALL-UNNAMED
```
