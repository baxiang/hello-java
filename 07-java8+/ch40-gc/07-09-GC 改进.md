# 31.9 GC 改进：G1、ZGC、Shenandoah

## 一、先问一个问题

**Java 有哪些垃圾收集器？**

### 生活例子：垃圾清理

```
Serial GC（单线程）
└── 一个人清理 → 慢

Parallel GC（多线程）
├── 多个人清理
└── 需要暂停（STW）

G1 GC（分区域）
├── 分区清理
└── 可预测停顿

ZGC/Shenandoah（并发）
├── 边运行边清理
└── 几乎不停顿
```

## 二、GC 演进

### Java 8：Parallel GC（默认）

```bash
# 启用 Parallel GC
-XX:+UseParallelGC

# 特点
# - 多线程
# - 需要 STW（Stop-The-World）
# - 适合吞吐量优先
```

### Java 11：G1 GC（默认）

```bash
# 启用 G1 GC
-XX:+UseG1GC

# 特点
# - 分区收集
# - 可预测停顿
# - 适合大堆内存
```

### Java 17+：ZGC（生产就绪）

```bash
# 启用 ZGC
-XX:+UseZGC

# 特点
# - 并发收集
# - 停顿时间<1ms
# - 支持 TB 级堆内存
```

## 三、G1 GC（Garbage First）

### 工作原理

```
堆内存划分
├── Region（区域）
│   ├── Eden（新生代）
│   ├── Survivor（幸存者）
│   └── Old（老年代）
└── Humongous（大对象）

收集过程
├── Young GC（只收集新生代）
└── Mixed GC（新生代 + 部分老年代）
```

### 配置参数

```bash
# 基本配置
-XX:+UseG1GC
-Xms4g -Xmx4g  # 堆大小

# 停顿时间目标
-XX:MaxGCPauseMillis=200  # 目标停顿时间（毫秒）

# Region 大小
-XX:G1HeapRegionSize=16m  # Region 大小（1-32MB）

# 并发线程
-XX:ConcGCThreads=2  # 并发线程数

# 大对象阈值
-XX:G1HeapWastePercent=5  # 浪费百分比
```

### 适用场景

```
✅ 适合：
- 大堆内存（>4GB）
- 需要可预测停顿
- 服务器应用

❌ 不适合：
- 小堆内存
- 实时性要求极高
```

## 四、ZGC（Z Garbage Collector）

### 工作原理

```
并发标记
├── 标记存活对象
└── 不暂停应用

并发整理
├── 移动存活对象
└── 不暂停应用

并发引用更新
├── 更新对象引用
└── 不暂停应用
```

### 配置参数

```bash
# 启用 ZGC
-XX:+UseZGC

# 堆大小
-Xms4g -Xmx4g

# 页面大小（Linux）
-XX:+UseTransparentHugePages  # 使用透明大页

# 日志
-Xlog:gc*:file=gc.log:time,uptime:filecount=5,filesize=10M
```

### 性能特点

| 特性 | G1 GC | ZGC |
|------|-------|-----|
| 停顿时间 | <200ms | <1ms |
| 堆大小 | 几 GB - 几十 GB | 几 GB - 几 TB |
| CPU 开销 | 低 | 中 |
| 吞吐量 | 高 | 略低 |

### 适用场景

```
✅ 适合：
- 低延迟要求
- 大堆内存
- 实时应用

❌ 不适合：
- 小堆内存
- CPU 资源紧张
```

## 五、Shenandoah GC

### 工作原理

```
类似 ZGC
├── 并发标记
├── 并发整理
└── 并发引用更新
```

### 配置参数

```bash
# 启用 Shenandoah
-XX:+UseShenandoahGC

# 堆大小
-Xms4g -Xmx4g

# 垃圾回收阈值
-XX:ShenandoahGCHeuristics=compact
```

### ZGC vs Shenandoah

| 特性 | ZGC | Shenandoah |
|------|-----|------------|
| 开发者 | Oracle | Red Hat |
| Java 版本 | Java 11+ | Java 12+ |
| 平台 | Linux/Windows/Mac | Linux/Windows |
| 性能 | 略优 | 略差 |

## 六、GC 日志分析

### 启用 GC 日志

```bash
# Java 9+
-Xlog:gc*:file=gc.log:time,uptime:filecount=5,filesize=10M

# Java 8
-XX:+PrintGCDetails
-XX:+PrintGCDateStamps
-Xloggc:gc.log
```

### GC 日志示例

```
[2024-01-15T14:30:00.000+0800] GC(1) Pause Young (Normal) (G1 Evacuation Pause)
[2024-01-15T14:30:00.100+0800] GC(1) Pause Young (Normal) (G1 Evacuation Pause) 100.000ms
```

### 分析工具

```bash
# GCViewer（可视化分析）
# 下载：https://github.com/chewiebug/GCViewer

# GCEasy（在线分析）
# 网站：https://gceasy.io/

# JFR（Java Flight Recorder）
jcmd <pid> JFR.start name=gc duration=60s filename=gc.jfr
```

## 七、GC 调优

### 调优步骤

```
1. 监控
   └── 收集 GC 日志

2. 分析
   └── 分析停顿时间、频率

3. 调整
   └── 调整堆大小、GC 参数

4. 验证
   └── 验证效果
```

### 常见问题和解决方案

```
问题 1：频繁 Full GC
原因：老年代空间不足
解决：
- 增加堆内存 -Xmx
- 调整新生代比例 -XX:NewRatio
- 检查内存泄漏

问题 2：停顿时间过长
原因：GC 算法不合适
解决：
- 改用 G1 GC -XX:+UseG1GC
- 改用 ZGC -XX:+UseZGC
- 调整停顿目标 -XX:MaxGCPauseMillis

问题 3：吞吐量低
原因：GC 太频繁
解决：
- 增加堆内存
- 减少对象创建
- 使用对象池
```

## 八、GC 选择指南

### 选择流程

```
1. 内存大小？
   ├── <1GB → Serial GC
   ├── 1-4GB → Parallel GC
   └── >4GB → G1/ZGC

2. 延迟要求？
   ├── 不敏感 → Parallel GC
   ├── <200ms → G1 GC
   └── <10ms → ZGC/Shenandoah

3. 堆大小？
   ├── <32GB → G1 GC
   └── >32GB → ZGC
```

### 推荐配置

```bash
# 小型应用（<1GB）
-XX:+UseSerialGC
-Xms512m -Xmx512m

# 中型应用（1-4GB）
-XX:+UseParallelGC
-Xms2g -Xmx2g

# 大型应用（>4GB）
-XX:+UseG1GC
-Xms4g -Xmx4g
-XX:MaxGCPauseMillis=200

# 低延迟应用
-XX:+UseZGC
-Xms8g -Xmx8g
```

## 九、小结

| GC | Java 版本 | 停顿时间 | 适用场景 |
|------|-----------|----------|----------|
| Serial | 所有 | 长 | 小型应用 |
| Parallel | 所有 | 中 | 吞吐量优先 |
| G1 | Java 7+ | <200ms | 大堆内存 |
| ZGC | Java 11+ | <1ms | 低延迟 |
| Shenandoah | Java 12+ | <10ms | 低延迟 |

**核心要点**：
- G1：大堆内存默认选择
- ZGC：低延迟首选
- Shenandoah：ZGC 替代方案

## 十、常见问题

### Q1：如何查看当前使用的 GC？

```bash
# 查看 GC
java -XX:+PrintCommandLineFlags -version

# 或
jstat -gc <pid>
```

### Q2：ZGC 和 G1 能一起用吗？

**不能**：
```bash
# 只能选择一个
-XX:+UseG1GC  # 或
-XX:+UseZGC   # 或
-XX:+UseShenandoahGC
```

### Q3：如何判断 GC 是否合适？

**监控指标**：
- 停顿时间
- GC 频率
- 吞吐量
- 堆内存使用率
