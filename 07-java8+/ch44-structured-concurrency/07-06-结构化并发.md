# 7.6 Structured Concurrency（结构化并发）

## 一、传统并发的缺陷

### 生活例子：餐厅厨房

想象一家餐厅的厨房：厨师长同时派了三个徒弟去准备菜品——切菜、腌肉、烧汤。如果烧汤的徒弟发现没有煤气了，传统做法下，厨师长不会通知另外两个徒弟停工，他们还在继续干活，最后菜切好了、肉腌好了，却做不出一道完整的菜。

这就是"无组织并发"的问题：任务之间没有关联，一个失败不影响其他任务继续运行，导致资源浪费和状态不一致。

"结构化并发"则是：厨师长派活时明确这三件事属于同一道菜，任何一环出问题，立即通知所有人停工，清理现场。

### 1.1 线程泄漏

```java
// 问题：如果 getUser 失败，getOrder 不会被取消
Future<User> user = executor.submit(() -> getUser(id));
Future<Order> order = executor.submit(() -> getOrder(id));
// 如果这里抛出异常，user 的线程还在运行！
return new Dashboard(user.get(), order.get());
```

传统 ExecutorService 提交任务后，主线程无法统一管理子任务的生命周期。一旦中间环节抛出异常，已提交的任务仍在后台运行，造成线程泄漏。

更严重的是，这些泄漏的线程可能持有数据库连接、文件句柄等有限资源，长时间积累会导致系统资源耗尽。手动管理 Future.cancel() 容易遗漏异常分支，代码变得臃肿且难以维护。

### 1.2 线程归属混乱

线程一旦提交到线程池，就脱离了创建者的控制范围。调试时无法回答以下问题：

- 这个线程是谁创建的？
- 它属于哪个业务请求？
- 它的父任务是什么？

结构化并发通过父子关系解决了线程归属问题，所有子任务的生命周期与作用域绑定，形成清晰的调用树。

## 二、StructuredTaskScope

### 2.1 基本概念

Java 21 预览特性，提供结构化并发编程模型。核心思想是：任务的生命周期绑定到代码块的作用域，进入作用域创建任务，退出作用域自动清理。

```java
try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
    Subtask<User> userTask = scope.fork(() -> getUser(id));
    Subtask<Order> orderTask = scope.fork(() -> getOrder(id));
    
    scope.join();           // 等待所有任务完成
    scope.throwIfFailed();  // 如果有失败则抛出异常
    
    return new Dashboard(userTask.get(), orderTask.get());
}
// 退出 try 块自动关闭所有任务
```

`StructuredTaskScope` 实现了 `AutoCloseable`，配合 try-with-resources 使用。fork 提交子任务，join 等待完成，throwIfFailed 统一处理错误。

注意：调用 fork 的线程必须是创建 scope 的同一线程，这是结构化并发的核心约束，确保任务树不会被破坏。子任务内部不能再 fork 新任务，如果需要嵌套结构，应创建子 scope。

## Q&A

**Q：StructuredTaskScope 是正式特性还是预览特性？**

A：Java 21 中作为预览特性引入，预计在 Java 25 成为正式特性。使用时需要添加 `--enable-preview` 参数。

**Q：能否在子任务中再次调用 fork？**

A：不能。fork 只能在创建 scope 的线程中调用，子任务内部调用会抛出 IllegalStateException。这是设计上的约束，保证任务层级结构清晰。

**Q：如何获取子任务的状态？**

A：通过 `Subtask.state()` 方法可以查看任务状态：`SUCCESS`、`FAILED`、`UNAVAILABLE`（未执行完成）。这在需要精细控制时非常有用。

### 2.2 ShutdownOnFailure vs ShutdownOnSuccess

| 策略 | 行为 | 适用场景 |
|------|------|----------|
| ShutdownOnFailure | 任一失败立即取消所有 | 必须全部成功 |
| ShutdownOnSuccess | 任一成功立即取消所有 | 多路冗余请求 |

ShutdownOnFailure 适用于"聚合型"场景，比如加载用户信息时需要同时获取基本资料、订单记录、偏好设置，缺一不可。

ShutdownOnSuccess 示例：

```java
try (var scope = new StructuredTaskScope.ShutdownOnSuccess<String>()) {
    scope.fork(() -> queryFromCache(id));
    scope.fork(() -> queryFromDatabase(id));
    scope.fork(() -> queryFromRemote(id));
    
    scope.join();
    return scope.result();  // 返回第一个成功的结果
}
```

多路冗余请求的典型场景：缓存命中则直接返回，不需要再查数据库和远程接口。

## 三、与虚拟线程配合

StructuredTaskScope 默认使用当前线程的 ForkJoinPool，更常见的是与虚拟线程池配合：

```java
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    executor.submit(() -> {
        try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
            scope.fork(() -> callServiceA());
            scope.fork(() -> callServiceB());
            scope.fork(() -> callServiceC());
            scope.join();
            scope.throwIfFailed();
        }
    }).get();
}
```

虚拟线程提供轻量级并发能力，结构化并发提供清晰的生命周期管理，两者结合是 Java 并发的最佳实践。

虚拟线程的优势在于创建和切换成本极低，一个请求可以轻松 fork 数十个子任务而不用担心平台线程耗尽。配合结构化并发后，这些子任务的生命周期完全受控，不会泄漏。

## 四、错误传播

```java
try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
    var task1 = scope.fork(() -> fetchUserProfile(userId));
    var task2 = scope.fork(() -> fetchUserOrders(userId));
    var task3 = scope.fork(() -> fetchUserSettings(userId));
    
    scope.join();
    scope.throwIfFailed();
    
    return new UserProfile(
        task1.get(), task2.get(), task3.get()
    );
} catch (ExecutionException e) {
    log.error("Failed to load user profile", e);
    throw new ServiceUnavailableException();
}
```

任一子任务失败时，throwIfFailed 会将异常包装为 ExecutionException 抛出，未完成的子任务会被自动取消。调用方只需捕获一次异常即可处理所有子任务的失败。

错误传播的完整流程：

1. 某个 fork 提交的任务抛出异常
2. ShutdownOnFailure 策略检测到失败，立即取消其他所有未完成的子任务
3. join() 返回，等待结束
4. throwIfFailed() 检查是否有失败，如果有则抛出 ExecutionException
5. 异常携带原始失败任务的信息，便于定位问题
6. try 块退出时，scope.close() 确保所有资源被释放

## 五、综合案例：电商详情页

```java
public ProductDetail getProductDetail(Long productId) {
    try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
        var product = scope.fork(() -> productService.getById(productId));
        var reviews = scope.fork(() -> reviewService.getByProduct(productId));
        var inventory = scope.fork(() -> inventoryService.getStock(productId));
        var price = scope.fork(() -> pricingService.getCurrentPrice(productId));
        
        scope.join();
        scope.throwIfFailed();
        
        return new ProductDetail(
            product.get(), reviews.get(), inventory.get(), price.get()
        );
    } catch (ExecutionException e) {
        throw new ProductDetailLoadException(productId, e);
    }
}
```

电商商品详情页需要聚合商品基础信息、用户评价、库存状态、实时价格四个接口。使用结构化并发后：

- 四个请求并行发起，减少总体响应时间
- 任一接口失败，其余请求自动取消，避免无效计算
- 作用域结束时所有资源自动释放，无泄漏风险

对比传统实现，代码从 "提交任务 → 逐个 get → 逐个异常处理 → 手动 cancel" 简化为 "fork → join → throwIfFailed"，可读性和可靠性大幅提升。

## 六、小结

| 概念 | 说明 |
|------|------|
| StructuredTaskScope | 结构化并发作用域 |
| fork() | 提交子任务 |
| join() | 等待所有任务完成 |
| ShutdownOnFailure | 失败时取消所有任务 |
| ShutdownOnSuccess | 成功时取消所有任务 |

### 对比传统并发

| 维度 | ExecutorService + Future | StructuredTaskScope |
|------|--------------------------|---------------------|
| 生命周期 | 手动管理 | 自动绑定作用域 |
| 错误处理 | 逐个检查 Future | 统一 throwIfFailed |
| 线程泄漏 | 需要手动 cancel | 作用域结束自动清理 |
| 任务归属 | 无法追溯创建者 | 清晰的父子关系树 |
| 资源释放 | 容易遗漏 | try-with-resources 保证 |
