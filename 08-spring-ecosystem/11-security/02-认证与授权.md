# 56.2 认证与授权：身份验证（生活类比：公司前台）

## 一、先问一个问题
**问题**：如何从数据库验证用户身份并分配权限？

### 生活例子
公司前台的工作流程：
- 访客报名字（用户输入用户名和密码）
- 前台查员工名单（从数据库查询用户信息）
- 发放不同颜色胸牌（根据角色分配不同权限）

自定义 UserDetailsService 就是这个"智能前台"——它不仅验证身份，还根据数据库中的角色信息决定你能访问哪些区域！

## 二、自定义 UserDetailsService

UserDetailsService 是 Spring Security 的核心接口，负责根据用户名加载用户详情和权限信息。

```java
@Service
public class CustomUserDetailsService implements UserDetailsService {
    
    @Autowired
    private UserMapper userMapper;
    
    @Override
    public UserDetails loadUserByUsername(String username) 
            throws UsernameNotFoundException {
        // 从数据库查询用户基本信息
        com.example.demo.entity.User user = 
            userMapper.findByUsername(username);
        
        if (user == null) {
            throw new UsernameNotFoundException("用户不存在: " + username);
        }
        
        // 查询用户拥有的角色
        List<String> roles = userMapper.findRolesByUserId(user.getId());
        List<GrantedAuthority> authorities = roles.stream()
            .map(role -> new SimpleGrantedAuthority("ROLE_" + role))
            .collect(Collectors.toList());
        
        // 构建 Spring Security 的 UserDetails 对象
        return new org.springframework.security.core.userdetails.User(
            user.getUsername(),
            user.getPassword(),
            user.isEnabled(),      // 账户是否启用
            true,                  // 账户是否过期
            true,                  // 凭证是否过期
            true,                  // 账户是否锁定
            authorities            // 权限列表
        );
    }
}
```

**关键说明**：
- `loadUserByUsername` 方法会被 Spring Security 在认证时自动调用
- 返回的 `UserDetails` 包含用户名、加密后的密码和权限列表
- 角色名称需要加 `ROLE_` 前缀，才能与 `hasRole()` 方法配合使用

## 三、安全配置

SecurityFilterChain 配置决定了哪些接口需要认证、哪些需要特定角色。

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {
    
    @Autowired
    private CustomUserDetailsService userDetailsService;
    
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            // 禁用 CSRF 保护（前后端分离的 API 项目不需要）
            .csrf(csrf -> csrf.disable())
            // 配置请求授权规则
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/auth/**").permitAll()          // 登录注册无需认证
                .requestMatchers("/api/admin/**").hasRole("ADMIN")    // 管理员接口
                .requestMatchers("/api/user/**").hasAnyRole("USER", "ADMIN")  // 用户或管理员
                .anyRequest().authenticated()                         // 其他接口需要认证
            )
            // 禁用表单登录（API 项目使用 Token 或 Basic 认证）
            .formLogin(form -> form.disable())
            // 启用 HTTP Basic 认证
            .httpBasic(Customizer.withDefaults());
        
        return http.build();
    }
    
    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }
}
```

**配置说明**：

| 配置项 | 作用 | 适用场景 |
|--------|------|----------|
| `csrf.disable()` | 禁用 CSRF 保护 | 无状态 API 项目 |
| `permitAll()` | 允许匿名访问 | 登录、注册、公开接口 |
| `hasRole("ADMIN")` | 需要 ADMIN 角色 | 管理后台接口 |
| `hasAnyRole()` | 需要任一角色 | 多角色共享接口 |

### 4.1 开启方法安全注解

### 4.2 使用 @PreAuthorize 保护方法

```java
@Service
public class UserService {
    
    // 只需已登录
    @PreAuthorize("isAuthenticated()")
    public User getCurrentUser() {
        // 获取当前登录用户信息
    }
    
    // 需要 ADMIN 角色
    @PreAuthorize("hasRole('ADMIN')")
    public List<User> getAllUsers() {
        // 查询所有用户列表
    }
    
    // 需要特定权限
    @PreAuthorize("hasAuthority('user:delete')")
    public void deleteUser(Long id) {
        // 删除指定用户
    }
    
    // 检查当前用户只能访问自己的数据
    @PreAuthorize("#userId == authentication.principal.id")
    public User getUserById(Long userId) {
        // 查询用户详情
    }
}
```

**常用 SpEL 表达式**：

| 表达式 | 含义 |
|--------|------|
| `isAuthenticated()` | 已登录 |
| `hasRole('ROLE')` | 拥有指定角色 |
| `hasAuthority('perm')` | 拥有指定权限 |
| `#userId == ...` | 方法参数与当前用户匹配 |

## 五、常见疑问

**Q1：UserDetailsService 的作用是什么？**
A：它是 Spring Security 加载用户信息的核心接口。认证时，框架会调用 `loadUserByUsername()` 方法，通过用户名从数据库查询用户详情和密码，然后与用户输入的密码进行比对。

**Q2：@EnableMethodSecurity 和 @EnableGlobalMethodSecurity 有什么区别？**
A：`@EnableMethodSecurity` 是 Spring Security 5.6+ 的新版注解，替代了旧的 `@EnableGlobalMethodSecurity`。新版支持更好的类型推断和更灵活的配置方式。新项目应优先使用 `@EnableMethodSecurity`。

**Q3：为什么角色要加 ROLE_ 前缀？**
A：Spring Security 的 `hasRole()` 方法内部会自动添加 `ROLE_` 前缀进行匹配。如果使用 `hasAuthority()` 则不需要前缀，两者可以按需选择。

## 六、小结

| 概念 | 说明 | 关键代码 |
|------|------|----------|
| UserDetailsService | 从数据源加载用户信息 | `loadUserByUsername()` |
| GrantedAuthority | 封装角色和权限 | `SimpleGrantedAuthority` |
| SecurityFilterChain | 配置接口访问规则 | `.authorizeHttpRequests()` |
| @PreAuthorize | 方法级权限控制 | SpEL 表达式 |
| BCryptPasswordEncoder | 密码加密器 | `new BCryptPasswordEncoder()` |
