# 13.1 微服务架构概述

## 一、架构演进

### 1.1 单体架构

```
┌─────────────────────────┐
│      单体应用            │
│  ┌─────────────────┐    │
│  │   用户模块       │    │
│  ├─────────────────┤    │
│  │   订单模块       │    │
│  ├─────────────────┤    │
│  │   商品模块       │    │
│  ├─────────────────┤    │
│  │   支付模块       │    │
│  └─────────────────┘    │
└─────────────────────────┘
```

**特点**：
- 所有功能在一个应用中
- 部署简单
- 开发效率高（初期）

**缺点**：
- 代码耦合度高
- 扩展性差
- 单点故障风险

### 1.2 微服务架构

```
                    ┌─────────────┐
                    │   Gateway   │
                    └──────┬──────┘
                           │
        ┌──────────────────┼──────────────────┐
        │                  │                  │
   ┌────▼────┐      ┌─────▼─────┐     ┌──────▼─────┐
   │ 用户服务 │      │  订单服务  │     │  商品服务   │
   └─────────┘      └───────────┘     └────────────┘
```

**特点**：
- 服务独立部署
- 技术栈灵活
- 易于扩展

**缺点**：
- 运维复杂
- 分布式事务
- 网络延迟

## 二、微服务优缺点

### 2.1 优点

| 优点 | 说明 |
|------|------|
| 独立部署 | 每个服务可独立发布 |
| 技术灵活 | 不同服务可用不同技术栈 |
| 易于扩展 | 按需扩展特定服务 |
| 故障隔离 | 单服务故障不影响整体 |
| 团队独立 | 小团队负责特定服务 |

### 2.2 挑战

| 挑战 | 说明 |
|------|------|
| 分布式复杂性 | 网络、延迟、容错 |
| 数据一致性 | 分布式事务处理 |
| 运维复杂度 | 需要自动化运维 |
| 服务治理 | 注册发现、配置管理 |
| 测试难度 | 集成测试复杂 |

## 三、Spring Cloud 简介

### 3.1 Spring Cloud 组件

```
Spring Cloud
├── 服务注册与发现：Nacos, Eureka
├── 配置中心：Nacos Config, Config Server
├── 服务调用：OpenFeign
├── 负载均衡：LoadBalancer, Ribbon
├── 熔断降级：Sentinel, Hystrix
├── API 网关：Gateway, Zuul
├── 链路追踪：Sleuth, Zipkin, SkyWalking
└── 分布式事务：Seata
```

### 3.2 Spring Cloud Alibaba

```
Spring Cloud Alibaba
├── Nacos：注册中心 + 配置中心
├── Sentinel：流量控制、熔断降级
├── Seata：分布式事务
├── RocketMQ：消息队列
└── Dubbo：RPC 框架
```

## 四、微服务核心概念

### 4.1 服务注册与发现

```
服务提供者 ──→ 注册中心 ←── 服务消费者
     ↑                           ↑
     └──────── 发现服务 ─────────┘
```

### 4.2 配置中心

```
┌──────────┐
│ 配置中心 │
└────┬─────┘
     │
     ├──→ 服务 A
     ├──→ 服务 B
     └──→ 服务 C
```

### 4.3 API 网关

```
客户端 → Gateway → 用户服务
                → 订单服务
                → 商品服务
```

### 4.4 熔断降级

```
客户端 → 服务 A → 服务 B
           ↓
        熔断器（正常/打开/半开）
```

## 五、微服务设计原则

### 5.1 单一职责

```
✅ 好：用户服务只负责用户相关
❌ 差：用户服务包含订单逻辑
```

### 5.2 服务自治

```
✅ 好：每个服务独立数据库
❌ 差：多个服务共享数据库
```

### 5.3 轻量级通信

```
✅ 好：RESTful API、gRPC
❌ 差：直接数据库访问
```

### 5.4 容错设计

```
✅ 好：熔断、降级、超时
❌ 差：无故障处理
```

## 六、微服务拆分策略

### 6.1 按业务领域拆分

```
电商系统
├── 用户服务
├── 商品服务
├── 订单服务
├── 支付服务
└── 库存服务
```

### 6.2 按功能拆分

```
├── 认证服务
├── 消息服务
├── 文件服务
└── 搜索服务
```

### 6.3 拆分粒度

```
过粗：大服务（接近单体）
适中：业务领域服务
过细：过多小服务（运维复杂）
```

## 七、综合示例

### 7.1 电商系统架构

```
                    ┌─────────────┐
                    │   Gateway   │
                    └──────┬──────┘
                           │
     ┌──────────┬──────────┼──────────┬──────────┐
     │          │          │          │          │
┌────▼───┐ ┌───▼────┐ ┌───▼───┐ ┌───▼───┐ ┌───▼───┐
│用户服务│ │商品服务│ │订单服务│ │支付服务│ │库存服务│
└────┬───┘ └───┬────┘ └───┬───┘ └───┬───┘ └───┬───┘
     │          │          │          │          │
     └──────────┴────┬─────┴──────────┴──────────┘
                     │
              ┌──────▼──────┐
              │    Nacos    │
              │ 注册中心 + 配置│
              └─────────────┘
```

### 7.2 服务调用链路

```
用户请求 → Gateway → 订单服务 → 用户服务
                          → 商品服务
                          → 支付服务
                          → 库存服务
```
