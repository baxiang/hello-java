# 递归方法

## 一、问题引入

你需要计算文件夹的总大小——文件夹里还有文件夹，层层嵌套，循环很难写。这类"自己包含自己"的结构，天然适合递归：遇到文件夹就递归进去，遇到文件就累加大小。

递归的核心思想：**把大问题拆成同类型的小问题，直到问题小到可以直接解决**。

> 生活类比：递归像俄罗斯套娃——打开最外层，里面是同样的套娃，一直拆到最小的那个（基准情况），再从最小的开始一层层回溯。

## 二、核心概念

### 递归两要素

1. **基准情况**（Base Case）：递归终止的条件，不调用自身
2. **递归情况**（Recursive Case）：调用自身，且必须向基准情况靠近

```java
返回值类型 方法(参数) {
    if (基准条件) return 直接结果;     // 终止
    return 递归表达式(更小的参数);      // 推进
}
```

缺少基准情况 → 无限递归 → `StackOverflowError`；不向基准靠近 → 同样无限递归。

## 三、代码对比

### 3.1 阶乘：递归 vs 循环

```java
// ✅ 递归：代码简洁，直接对应数学定义 n! = n × (n-1)!
public static int factorial(int n) {
    if (n == 1) return 1;       // 基准情况
    return n * factorial(n - 1); // 递归情况
}

// ✅ 循环：性能更好，无栈溢出风险
public static int factorial(int n) {
    int result = 1;
    for (int i = 2; i <= n; i++) result *= i;
    return result;
}
```

执行过程（递归展开再回溯）：
```
factorial(5)
= 5 * factorial(4)
= 5 * 4 * factorial(3)
= 5 * 4 * 3 * factorial(2)
= 5 * 4 * 3 * 2 * factorial(1)  ← 基准情况
= 5 * 4 * 3 * 2 * 1
= 120                            ← 回溯得出结果
```

### 3.2 斐波那契：朴素递归的陷阱

```java
// ❌ 朴素递归：指数级重复计算，fib(50) 就卡死
public static int fib(int n) {
    if (n <= 1) return n;
    return fib(n - 1) + fib(n - 2);  // fib(5) 中 fib(3) 被计算 2 次
}

// ✅ 记忆化：用数组缓存已计算的结果，避免重复
static int[] memo = new int[100];
public static int fib(int n) {
    if (n <= 1) return n;
    if (memo[n] != 0) return memo[n];  // 关键：命中缓存直接返回
    memo[n] = fib(n - 1) + fib(n - 2);
    return memo[n];
}
```

## 四、实现方式

### 4.1 经典递归问题

**阶乘**

```java
public static int factorial(int n) {
    if (n <= 1) return 1;
    return n * factorial(n - 1);
}
factorial(5);  // 输出: 120
```

**斐波那契数列**（记忆化版本）

```java
static int[] memo = new int[100];
public static int fib(int n) {
    if (n <= 1) return n;
    if (memo[n] != 0) return memo[n];
    memo[n] = fib(n - 1) + fib(n - 2);
    return memo[n];
}
fib(10);  // 输出: 55
```

**字符串反转**

```java
public static String reverse(String s) {
    if (s.isEmpty()) return s;              // 基准：空串
    return reverse(s.substring(1)) + s.charAt(0); // 取首字符放末尾
}
reverse("hello");  // 输出: olleh
```

### 4.2 尾递归

递归调用是方法最后一步操作，无需回溯处理：

```java
// 普通递归：return n * factorial(n-1)，回溯时还要乘 n
// 尾递归：结果通过参数传递，无需回溯
public static int factorial(int n, int acc) {
    if (n <= 1) return acc;
    return factorial(n - 1, acc * n);  // 关键：递归调用是最后一步
}
factorial(5, 1);  // 输出: 120
```

> 注意：Java 不做尾递归优化，尾递归仍会栈溢出。但尾递归写法更易转化为循环。

### 4.3 递归与循环的选择

| 场景 | 推荐 | 原因 |
|------|------|------|
| 简单重复（求和、遍历） | 循环 | 性能好，无栈溢出风险 |
| 树/图遍历、分治 | 递归 | 代码更清晰，循环实现复杂 |
| 递归深度可控 | 均可 | 看可读性 |
| 递归深度不可控 | 循环 | 避免 StackOverflowError |

## 五、Q&A

**Q：递归一定比循环慢吗？**
A：有函数调用开销，理论上更慢。但现代 JVM 会内联简单递归，实际差距不大。真正慢的是重复计算（如朴素斐波那契），用记忆化即可解决。

**Q：什么时候必须用递归？**
A：没有"必须"，所有递归都能改写成循环。但树遍历、回溯算法等场景，递归代码简洁得多，循环实现需要手动维护栈。

**Q：如何避免 StackOverflowError？**
A：① 确保有基准情况且向其靠近；② 控制递归深度；③ 深度大时改用循环或尾递归。

**Q：尾递归有什么用？**
A：尾递归的递归调用是最后一步，理论上不需要保存调用栈。部分语言（Scala、Haskell）会自动优化为循环，但 Java 不会，所以尾递归在 Java 中仍可能栈溢出。

## 六、小结表格

| 要点 | 内容 |
|------|------|
| 递归要素 | 基准情况（终止）+ 递归情况（推进） |
| 缺少基准情况 | 导致 StackOverflowError |
| 朴素递归陷阱 | 重复计算，指数级时间复杂度 |
| 记忆化优化 | 缓存已计算结果，避免重复 |
| 尾递归 | 递归调用是最后一步，易转化为循环 |
| Java 尾递归优化 | 不支持，仍可能栈溢出 |
| 递归 vs 循环 | 递归代码简洁，循环性能稳定 |
| 适用场景 | 树/图遍历、分治、回溯 |
