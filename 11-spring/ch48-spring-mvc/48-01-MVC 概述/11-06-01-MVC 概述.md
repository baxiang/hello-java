# 48.1 MVC 概述：什么是 MVC 模式？

## 一、先问一个问题

**什么是 MVC？为什么需要它？**

### 生活例子：餐厅

```
顾客（用户）
  ↓
点餐（请求）
  ↓
服务员（Controller）
  ↓
厨房（Model）
  ↓
做菜（处理业务）
  ↓
服务员上菜（View）
```

**MVC 就是**：
- 服务员（Controller）：接待顾客，安排订单
- 厨房（Model）：做菜，处理业务
- 上菜（View）：把菜端给顾客

## 二、没有 MVC 会怎样

### 例子：小摊贩

```
顾客 → 老板点餐 → 老板做菜 → 老板上菜
```

**问题**：
- 老板一个人干所有事
- 忙不过来
- 代码混乱，难以维护

### 代码例子（没有 MVC）

```java
// 所有代码写在一起
public class UserService {
    
    public void handleRequest(HttpServletRequest request, 
                             HttpServletResponse response) {
        // 1. 接收参数
        String username = request.getParameter("username");
        String password = request.getParameter("password");
        
        // 2. 业务逻辑
        User user = new User(username, password);
        userDao.save(user);
        
        // 3. 返回响应
        response.getWriter().println("注册成功");
    }
}
```

**问题**：
- 接收参数、业务逻辑、返回响应混在一起
- 难以测试
- 难以修改

## 三、MVC 的好处

### 例子：正规餐厅

```
顾客 → 服务员 → 厨房 → 服务员 → 顾客
       (C)      (M)     (V)
```

**好处**：
- 分工明确
- 各司其职
- 易于维护和扩展

### 代码例子（有 MVC）

```java
// Controller：只负责接收请求和返回响应
@Controller
public class UserController {
    
    @Autowired
    private UserService userService;
    
    @PostMapping("/register")
    public String register(@RequestParam String username,
                          @RequestParam String password) {
        // 只负责调用 Service
        userService.register(username, password);
        return "success";
    }
}

// Service：只负责业务逻辑
@Service
public class UserService {
    
    @Autowired
    private UserDao userDao;
    
    public void register(String username, String password) {
        // 业务逻辑
        User user = new User(username, password);
        userDao.save(user);
    }
}

// View：只负责显示
// success.jsp 或 success.html
```

**好处**：
- Controller 只管接收请求
- Service 只管业务逻辑
- View 只管显示
- 各司其职，互不干扰

## 四、MVC 的三个角色

### Model（模型）

**生活例子**：厨房

**作用**：
- 处理业务逻辑
- 操作数据库
- 返回数据

**代码**：
```java
@Service
public class UserService {
    
    public User getUserById(Long id) {
        return userDao.findById(id);
    }
}
```

### View（视图）

**生活例子**：上菜

**作用**：
- 显示数据
- 渲染页面
- 返回响应

**代码**：
```html
<!-- success.html -->
<html>
<body>
    <h1>注册成功！</h1>
</body>
</html>
```

### Controller（控制器）

**生活例子**：服务员

**作用**：
- 接收请求
- 调用 Model
- 返回 View

**代码**：
```java
@Controller
public class UserController {
    
    @Autowired
    private UserService userService;
    
    @GetMapping("/user/{id}")
    public String getUser(@PathVariable Long id, Model model) {
        User user = userService.getUserById(id);
        model.addAttribute("user", user);
        return "user-detail";
    }
}
```

## 五、Spring MVC 的工作流程

### 完整流程

```
1. 用户发送请求
   ↓
2. DispatcherServlet（前端控制器）
   ↓
3. HandlerMapping（查找 Controller）
   ↓
4. Controller（处理请求）
   ↓
5. ModelAndView（模型和视图）
   ↓
6. ViewResolver（解析视图）
   ↓
7. View（渲染页面）
   ↓
8. 返回响应
```

### 生活例子理解

```
1. 顾客到餐厅（用户请求）
   ↓
2. 前台接待（DispatcherServlet）
   ↓
3. 安排服务员（HandlerMapping）
   ↓
4. 服务员点餐（Controller）
   ↓
5. 厨房做菜（Model）
   ↓
6. 服务员确认菜品（ModelAndView）
   ↓
7. 装盘（ViewResolver）
   ↓
8. 上菜（View）
   ↓
9. 顾客用餐（返回响应）
```

## 六、核心组件

### DispatcherServlet（前端控制器）

**作用**：
- 统一入口
- 分发请求
- 协调各个组件

**配置**：
```java
// Spring Boot 自动配置，无需手动配置
```

### HandlerMapping（处理器映射）

**作用**：
- 根据 URL 找 Controller
- 就像：根据菜品找厨师

**代码**：
```java
@RequestMapping("/user")
public class UserController {
    // ...
}

// 访问 /user 时，找到 UserController
```

### ViewResolver（视图解析器）

**作用**：
- 根据逻辑视图名找真实页面
- 就像：根据菜名找盘子

**配置**：
```java
// application.properties
spring.thymeleaf.prefix=classpath:/templates/
spring.thymeleaf.suffix=.html

// Controller 返回 "success"
// 实际查找：classpath:/templates/success.html
```

## 七、常见疑问

### Q1：MVC 和三层架构有什么区别？

**三层架构**：
```
Controller → Service → Dao
```

**MVC**：
```
Controller → Model → View
```

**关系**：
- MVC 是表现层的模式
- 三层是整体架构
- 可以结合使用

### Q2：Spring MVC 和 Spring Boot 有什么区别？

**Spring MVC**：
- Web 框架
- 需要手动配置

**Spring Boot**：
- 快速开发框架
- 自动配置 Spring MVC
- 开箱即用

### Q3：什么时候用 MVC？

**适合场景**：
- Web 应用
- 需要页面展示
- 需要 API 接口

**不适合**：
- 简单的脚本
- 批处理程序
- 后台任务

## 八、小结

| 角色 | 生活类比 | 作用 |
|------|----------|------|
| Model | 厨房 | 处理业务逻辑 |
| View | 上菜 | 显示数据 |
| Controller | 服务员 | 接收请求、协调 |
| DispatcherServlet | 前台接待 | 统一入口 |
| ViewResolver | 装盘 | 解析视图 |

**MVC 的好处**：
- 分工明确
- 易于维护
- 易于测试
- 易于扩展
