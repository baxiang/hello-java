# 访问控制与单例

## 一、问题引入

一个银行账户类，余额可以直接被外部修改吗？

❌ **公开余额字段**：

```java
class Account {
    double balance; // ❌ 任何人都能改
}
account.balance = -1000000; // 危险！
```

✅ **用 private 保护，只提供有限方法**：

```java
class Account {
    private double balance;
    public void deposit(double amount) {
        if (amount > 0) balance += amount;
    }
    public void withdraw(double amount) {
        if (amount > 0 && amount <= balance) balance -= amount;
    }
}
```

访问控制保护数据安全，单例保证全局唯一——两者都体现了**"控制访问"**的核心思想。

## 二、核心概念

### 2.1 四种访问级别

| 修饰符 | 同类 | 同包 | 子类 | 其他包 |
|--------|------|------|------|--------|
| `private` | ✅ | ❌ | ❌ | ❌ |
| (default) | ✅ | ✅ | ❌ | ❌ |
| `protected` | ✅ | ✅ | ✅ | ❌ |
| `public` | ✅ | ✅ | ✅ | ✅ |

> **类比**：访问控制就像**保险柜的权限**——private = 只有自己知道密码；default = 同办公室的人能开；protected = 自己和家人能开；public = 大门敞开谁都能进。

### 2.2 实战选型原则

```
最小权限原则：能用 private 就不 default，能用 default 就不 protected，能用 protected 就不 public。

成员变量 → private（隐藏数据）
需要子类访问 → protected
真正公开 API → public
只在包内共享 → default
```

### 2.3 protected 的特殊场景

同一个包外的子类**可以继承** protected 成员，但通过**父类实例**访问时不行：

```java
package com.pkg1;
public class Parent {
    protected int value = 10;
}

package com.pkg2;
class Child extends Parent {
    void test() {
        System.out.println(value);            // ✅ 继承来的，可以访问
        Parent p = new Parent();
        // System.out.println(p.value);        // ❌ 通过父类实例，不行
    }
}
```

### 2.4 为什么访问控制和单例放在一起

访问控制限制"谁能访问"，单例限制"能创建几个"——都是对对象访问权限的控制。单例用 private 构造方法阻止外部 new，正是访问控制思想的典型应用。

## 三、代码对比

❌ **所有成员 public**（等于裸奔）：

```java
class User {
    public String password; // ❌ 任何人都能读
    public void withdraw(double amount) {
        balance -= amount; // ❌ 没检查
    }
}
```

✅ **遵循最小权限原则**：

```java
class User {
    private String password;
    public void withdraw(double amount) {
        if (amount <= 0) throw new IllegalArgumentException();
        if (amount > balance) throw new IllegalArgumentException("余额不足");
        balance -= amount;
    }
}
```

❌ **懒汉式不加同步（多线程不安全）**：

```java
private static Singleton instance;
public static Singleton getInstance() {
    if (instance == null) {
        instance = new Singleton(); // ❌ 多线程下可能创建多个实例
    }
    return instance;
}
```

✅ **加 DCL 或用静态内部类**（见下方实现方式）

## 四、实现方式

### 4.1 饿汉式（类加载时创建）

```java
class Singleton {
    private static final Singleton INSTANCE = new Singleton();
    private Singleton() { }
    public static Singleton getInstance() { return INSTANCE; }
}
```

优点：线程安全，实现简单。缺点：类加载时就创建，可能浪费资源。

### 4.2 懒汉式 + DCL（双重检查锁定）

```java
class Singleton {
    private static volatile Singleton instance;
    private Singleton() { }

    public static Singleton getInstance() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null) {
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```

> **为什么必须用 volatile？** `new Singleton()` 不是原子操作，分为三步：① 分配内存 → ② 初始化对象 → ③ 引用指向内存。JVM 可能重排为 ①→③→②，导致其他线程拿到未初始化的对象。`volatile` 禁止指令重排序，保证顺序。

### 4.3 静态内部类（推荐）

```java
class Singleton {
    private Singleton() { }
    private static class Holder {
        private static final Singleton INSTANCE = new Singleton();
    }
    public static Singleton getInstance() {
        return Holder.INSTANCE; // 第一次调用时加载，延迟初始化
    }
}
```

### 4.4 枚举（最简洁，推荐）

```java
enum Singleton {
    INSTANCE;
    public void doSomething() { }
}
Singleton.INSTANCE.doSomething();
```

枚举单例天然防止反射和序列化攻击——这是枚举单例的最大优势（Joshua Bloch 推荐）。

## 五、Q&A

**Q：单例模式能用反射破坏吗？**
A：普通单例可以（反射调用私有构造）。枚举单例天然防止反射和序列化攻击。

**Q：protected 和 public 什么时候用？**
A：protected 用于"希望子类能用但不想对外公开"的成员；public 用于真正需要对外开放的 API。

**Q：四种单例怎么选？**
A：不需要延迟初始化 → 饿汉式；需要延迟初始化 → 静态内部类；需要防反射 → 枚举；DCL 了解即可，实际中优先选静态内部类或枚举。

## 六、小结

| 要点 | 说明 |
|------|------|
| 最小权限原则 | 优先用最严格的访问级别 |
| protected 特殊场景 | 同包外子类不能通过父类实例访问 |
| private 构造 | 单例的核心：阻止外部 new |
| 饿汉式单例 | 类加载时创建，线程安全 |
| DCL 单例 | volatile 禁止指令重排序，保证线程安全 |
| 静态内部类单例 | 延迟初始化，推荐 |
| 枚举单例 | 最简洁，防止反射和序列化攻击 |
