# 访问控制与设计模式

## 一、问题引入

一个银行账户类，余额可以直接被外部修改吗？

❌ **公开余额字段**：

```java
class Account {
    double balance; // ❌ 任何人都能改
}
account.balance = -1000000; // 危险！
```

✅ **用 private 保护，只提供有限方法**：

```java
class Account {
    private double balance;
    public void deposit(double amount) {
        if (amount > 0) balance += amount; // ✅ 检查后操作
    }
    public void withdraw(double amount) {
        if (amount > 0 && amount <= balance) balance -= amount; // ✅ 检查后操作
    }
}
```

这就是**访问控制**的价值：用最小可见性保护数据安全。

## 二、核心概念

### 2.1 四种访问级别

| 修饰符 | 同类 | 同包 | 子类 | 其他包 |
|--------|------|------|------|--------|
| `private` | ✅ | ❌ | ❌ | ❌ |
| (default) | ✅ | ✅ | ❌ | ❌ |
| `protected` | ✅ | ✅ | ✅ | ❌ |
| `public` | ✅ | ✅ | ✅ | ✅ |

### 2.2 实战选型原则

```
最小权限原则：能用 private 就不 default，能用 default 就不 protected，能用 protected 就不 public。

成员变量 → private（隐藏数据）
需要子类访问 → protected
真正公开 API → public
只在包内共享 → default
```

### 2.3 protected 的特殊场景

同一个包外的子类**可以继承** protected 成员，但通过**父类实例**访问时不行：

```java
package com.pkg1;
public class Parent {
    protected int value = 10;
}

package com.pkg2;
class Child extends Parent {
    void test() {
        System.out.println(value);            // ✅ 继承来的，可以访问
        Parent p = new Parent();
        // System.out.println(p.value);        // ❌ 通过父类实例，不行
    }
}
```

### 2.4 类的访问控制

- **顶层类**（不在其他类内部）：只能用 `public` 或 default
- **内部类**：四种都可以用

## 三、单例设计模式

### 3.1 问题引入

全局只需要一个对象（比如配置管理器、日志管理器）时，怎么保证只有一个实例？

### 3.2 饿汉式（类加载时创建）

```java
class Singleton {
    private static final Singleton INSTANCE = new Singleton();
    private Singleton() { }
    public static Singleton getInstance() { return INSTANCE; }
}
```

优点：线程安全，实现简单。缺点：类加载时就创建，可能浪费资源。

### 3.3 懒汉式 + DCL（双重检查锁定）

```java
class Singleton {
    private static volatile Singleton instance;
    private Singleton() { }

    public static Singleton getInstance() {
        if (instance == null) {            // 第一次检查
            synchronized (Singleton.class) {
                if (instance == null) {   // 第二次检查
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```

### 3.4 静态内部类（推荐）

```java
class Singleton {
    private Singleton() { }
    private static class Holder {
        private static final Singleton INSTANCE = new Singleton();
    }
    public static Singleton getInstance() {
        return Holder.INSTANCE; // 第一次调用时加载，延迟初始化
    }
}
```

### 3.5 枚举（最简洁，推荐）

```java
enum Singleton {
    INSTANCE;
    public void doSomething() { }
}
Singleton.INSTANCE.doSomething();
```

## 四、代码对比

❌ **所有成员 public**（等于裸奔）：

```java
class User {
    public String password; // ❌ 任何人都能读
    public void withdraw(double amount) {
        balance -= amount; // ❌ 没检查
    }
}
```

✅ **遵循最小权限原则**：

```java
class User {
    private String password; // ✅ 私有
    public String getUsername() { return username; } // ✅ 适度开放
    public void withdraw(double amount) {
        if (amount <= 0) throw new IllegalArgumentException();
        if (amount > balance) throw new IllegalArgumentException("余额不足");
        balance -= amount;
    }
}
```

❌ **懒汉式不加同步（多线程不安全）**：

```java
private static Singleton instance;
public static Singleton getInstance() {
    if (instance == null) {
        instance = new Singleton(); // ❌ 多线程下可能创建多个实例
    }
    return instance;
}
```

✅ **加 DCL 或用静态内部类**：

```java
// DCL
private static volatile Singleton instance;
public static Singleton getInstance() {
    if (instance == null) {
        synchronized (Singleton.class) {
            if (instance == null) instance = new Singleton();
        }
    }
    return instance;
}

// 或静态内部类
private static class Holder {
    private static final Singleton INSTANCE = new Singleton();
}
```

## 五、Q&A

**Q：为什么 String 是 final 的？**
A：为了安全性（不可变）、线程安全、可以缓存 hashCode、支持字符串常量池复用。

**Q：单例模式能用反射破坏吗？**
A：普通单例可以（反射调用私有构造）。枚举单例天然防止反射和序列化攻击——这是枚举单例的最大优势。

**Q：protected 和 public 什么时候用？**
A：protected 用于"希望子类能用但不想对外公开"的成员；public 用于真正需要对外开放的 API。

**Q：内部类可以用四种访问修饰符中的哪些？**
A：都可以。成员内部类、局部内部类、匿名内部类都可以用 private/protected/default/public 修饰。

## 六、小结

| 要点 | 说明 |
|------|------|
| private | 最严格，只在本类内可见 |
| default | 包内可见 |
| protected | 同包 + 子类可见 |
| public | 完全开放 |
| 最小权限原则 | 优先用最严格的访问级别 |
| 饿汉式单例 | 类加载时创建，线程安全 |
| DCL 单例 | 延迟初始化，线程安全 |
| 静态内部类单例 | 延迟初始化，推荐 |
| 枚举单例 | 最简洁，防止反射和序列化攻击 |
