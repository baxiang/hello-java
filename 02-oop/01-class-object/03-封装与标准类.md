# 封装与标准类设计

## 一、问题引入

❌ **属性公开等于裸奔**：

```java
class Person {
    public String name;
    public int age;
}

Person p = new Person();
p.name = "张三";
p.age = -100;     // ❌ 危险！年龄可以是负数吗？
p.age = 300;      // ❌ 300 岁的人存在吗？
```

✅ **用 private 封起来，通过方法有限度地开放**：

```java
class Person {
    private String name;
    private int age;

    public void setAge(int age) {
        if (age >= 0 && age <= 150) { // ✅ 校验，拒绝无效值
            this.age = age;
        }
    }
}
```

**封装的核心**：把数据藏起来，只暴露必要的操作入口，在入口处加校验。

## 二、核心概念

### 2.1 访问权限修饰符

| 修饰符 | 同类 | 同包 | 子类 | 其他包 |
|--------|------|------|------|--------|
| private | ✅ | ❌ | ❌ | ❌ |
| (default) | ✅ | ✅ | ❌ | ❌ |
| protected | ✅ | ✅ | ✅ | ❌ |
| public | ✅ | ✅ | ✅ | ✅ |

> **类比**：封装就像 **ATM 机**——你不能直接打开机器数钱（private），只能通过屏幕和按钮（public 方法）操作，且每次取款都有额度校验（setter 里的条件判断）。

**实战原则**：成员变量通常 `private`，只在本类内访问；需要子类使用但不对外开放时用 `protected`；真正要公开的才用 `public`。

### 2.2 getter / setter

| 场景 | 方法命名 |
|------|----------|
| 普通类型 | `getName()` / `setName()` |
| 布尔类型 | `isMarried()` / `setMarried()` |
| 只读属性 | 只提供 getter，不提供 setter |
| 只写属性 | 只提供 setter，不提供 getter（少见，如密码） |

```java
private boolean married;

public boolean isMarried() { return married; } // 注意：is 开头
public void setMarried(boolean married) { this.married = married; }
```

### 2.3 为什么要封装而不直接公开

公开字段意味着外部可以任意赋值，校验逻辑无处安放。封装后，所有赋值必须经过 setter，校验集中在一处，维护成本大幅降低。这也是**最小权限原则**的体现——不该暴露的不暴露。

## 三、代码对比

❌ **直接返回内部数组引用**——外部可以修改数组：

```java
class Data {
    private int[] data = {1, 2, 3};

    public int[] getData() {
        return data; // ❌ 外部拿到数组引用，可以修改内部数据
    }
}
```

✅ **返回副本或只读视图**：

```java
public int[] getData() {
    return data.clone(); // ✅ 返回副本，修改不影响内部
}
```

❌ **不在 setter 中验证**：

```java
public void setAge(int age) {
    this.age = age; // ❌ 无效值长驱直入
}
```

✅ **在 setter 中加防御**：

```java
public void setAge(int age) {
    if (age < 0 || age > 150) {
        throw new IllegalArgumentException("年龄必须在 0-150 之间");
    }
    this.age = age;
}
```

## 四、实现方式

### 4.1 标准 JavaBean 四件套

```java
public class User implements Serializable {
    private static final long serialVersionUID = 1L; // 版本号，防止反序列化出错
    private Long id;
    private String username;

    public User() { }                                      // 无参构造
    public User(Long id, String username) { this.id = id; this.username = username; }

    public Long getId() { return id; }
    public void setId(Long id) { this.id = id; }
    public String getUsername() { return username; }
    public void setUsername(String username) { this.username = username; }
}
```

> `serialVersionUID` 是序列化版本号。不写时 JVM 会自动生成，但类结构变化后反序列化会失败。显式声明可以控制兼容性。

### 4.2 Lombok 简化（现代开发常用）

```java
@Data // 自动生成 getter/setter/toString/equals/hashCode/无参构造
public class User {
    private Long id;
    private String username;
}
```

`@Data` 是 Lombok 注解，编译时自动生成四件套代码。源码简洁，但需注意：`@Data` 生成的 `equals/hashCode` 包含所有字段，有时需要用 `@EqualsAndHashCode(of = {"id"})` 精确控制。

### 4.3 构造方法中调用 setter（利用校验）

```java
public class Person {
    private int age;

    public Person() { }
    public Person(int age) {
        setAge(age); // ✅ 利用 setter 的校验
    }

    public void setAge(int age) {
        if (age < 0) throw new IllegalArgumentException("年龄不能为负");
        this.age = age;
    }
}
```

### 4.4 JavaBean 应用场景

Java 很多框架（Spring、JSP、Jackson）都要求类符合 JavaBean 规范才能使用：
- 序列化（JSON 转换）
- 依赖注入
- EL 表达式

## 五、Q&A

**Q：所有字段都要 private 吗？**
A：大多数情况下是的。`private` 是最小权限原则的体现——除非有充分理由对外开放，否则默认 private。

**Q：布尔字段的 getter 命名用 get 还是 is？**
A：标准做法是 `isXxx()`（如 `isEmpty()`）。IDE 自动生成时也会这样处理。

**Q：计算属性（如矩形的长×宽=面积）需要字段吗？**
A：不需要。定义一个**只读 getter** 即可：

```java
public double getArea() {
    return width * height; // 不存字段，用时计算
}
```

**Q：为什么要实现 Serializable？**
A：标记接口，告诉 JVM 这个类的对象可以被序列化（转成字节流）。用于网络传输、持久化到文件、分布式 Session 存储等场景。

## 六、小结

| 要点 | 说明 |
|------|------|
| private 封装 | 成员变量默认 private，通过 getter/setter 有限度开放 |
| setter 校验 | 在入口处拒绝无效值，比在业务逻辑里散落校验更好 |
| JavaBean | 无参构造 + private 字段 + getter/setter + Serializable |
| serialVersionUID | 显式声明，防止类结构变化后反序列化失败 |
| Lombok @Data | 编译时生成四件套，注意 equals/hashCode 精确控制 |
| 计算属性 | 只提供 getter，不存字段 |
| 返回副本 | 引用类型 getter 返回副本，防止外部意外修改 |
