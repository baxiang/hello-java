## 写作规范

### 基本要求

1. **每节 150-250 行**，按功能模块拆分
2. 代码示例只保留核心部分，保持精简
3. 教程语言为**中文**，新增内容使用中文编写
4. 不生成可执行代码，只编写教程文档
5. 每节只围绕一个主问题展开；如果出现两个并列主题，应拆分成两节，或明确说明二者的因果关系
6. 每节至少包含 1 个生活类比、1 个真实开发场景、1 个易错边界；抽象概念必须落到可感知的例子上
7. 每节必须回答至少 1 个"为什么这样设计"，避免只罗列 API 和结论

### 文节结构（六段式）

借鉴《Effective Java》问题驱动 + 《Core Java》渐进展开 + 《Head First Java》对话互动：

```
一、问题引入          ← 真实痛点 + 反直觉现象 + 本节要解决的问题
二、核心概念          ← 定义 + 分类 + 生活类比 + 设计原因
三、代码对比          ← 反面写法 ❌ → 推荐写法 ✅ → 为什么更好
四、实现方式          ← 标准写法 + 常见变体 + 适用场景 + 边界条件
五、Q&A              ← 易错点、面试追问、工程实践中的常见疑问
六、小结表格          ← 概念、用法、适用场景、常见坑的结构化速查
```

六段式是**硬性结构**，一级小节标题必须严格对应上述六项：

- `## 一、问题引入`
- `## 二、核心概念`
- `## 三、代码对比`
- `## 四、实现方式`
- `## 五、Q&A`
- `## 六、小结`

不得把"内置 API 列表"、"设计模式"、"扩展知识"等内容直接放成 `三、` 或 `四、`，导致六段式错位。扩展知识应放入对应段落内部，例如"二、核心概念"中的子标题，或"四、实现方式"中的变体说明。

### 六段式细化要求

#### 一、问题引入

- 先写读者会遇到的具体问题，不先抛定义
- 最好包含一个错误现象、低效写法或维护痛点
- 结尾明确本节目标：学完能解决什么问题

#### 二、核心概念

- 先给一句话定义，再展开分类和边界
- 必须包含一个生活类比，但类比不能代替技术解释
- 必须说明至少一个"为什么"：为什么 Java 这样设计、为什么推荐这种用法、为什么这个限制存在
- 对容易混淆的概念要用表格对比

#### 三、代码对比

- 至少包含一组 `❌ 错误/低质量写法` 与 `✅ 推荐写法`
- 对比后必须解释改进点：安全性、可读性、扩展性、性能或维护成本
- 代码只展示当前知识点相关部分，不堆叠无关样板

#### 四、实现方式

- 按"基础写法 → 常见场景 → 进阶变体 → 使用边界"递进
- 优先讲 JDK 标准库和语言原生能力，再补充第三方工具或框架
- 涉及并发、IO、反射、序列化、数据库、网络等主题时，先讲正确性和安全性，再讲性能优化

#### 五、Q&A

- 至少 3 个问题，覆盖初学者易错点、面试追问、生产实践陷阱
- 回答要短而准，必要时补一个最小代码片段
- 避免重复前文定义，重点处理边界和误区

#### 六、小结

- 必须使用表格
- 表格至少包含"要点"和"说明"两列
- 对复杂主题建议增加"适用场景"、"常见坑"、"推荐做法"列

### 写作原则

借鉴经典 Java 书籍的核心理念：

| 原则 | 来源 | 说明 |
|------|------|------|
| **问题先行** | Effective Java | 每节以真实问题/痛点开篇，不先抛概念定义 |
| **对比教学** | Effective Java | 先展示常见错误写法（❌），再给出推荐写法（✅），说明原因 |
| **渐进深入** | Core Java | 从简单用法 → 常见场景 → 高级特性，逐层递进 |
| **理解本质** | Thinking in Java | 不仅说"怎么做"，更要说"为什么这样设计" |
| **安全优先** | Java Concurrency in Practice | 并发/API 相关内容，先讲安全性再讲性能 |
| **类比映射** | Head First Java | 用生活类比解释抽象概念，但类比之后必须回到严谨定义 |
| **案例驱动** | 深入理解 JVM | 用生产级场景/面试真题佐证，避免纯理论 |
| **重构思维** | 重构 | 展示代码演进过程：初始写法 → 发现问题 → 逐步改进 |

### 技术文章质量标准

每篇文章写完后按以下标准自检，目标分不低于 **8.5 / 10**：

| 维度 | 权重 | 达标标准 |
|------|------|----------|
| 结构规范性 | 20% | 六段式标题准确，不错位；行数 150-250 行 |
| 内容准确性 | 20% | 概念、代码、边界条件无误导；术语前后一致 |
| 问题驱动 | 15% | 开头有真实痛点，全文围绕问题展开 |
| 对比教学 | 15% | 至少一组反面写法与推荐写法，且说明原因 |
| 理解本质 | 10% | 解释设计动机、底层约束或工程取舍 |
| 渐进深入 | 10% | 从基础用法到常见场景再到边界条件 |
| 类比与案例 | 5% | 至少一个贴切类比和一个开发场景 |
| 表达与精简 | 5% | 语言清晰，代码最小化，无明显重复 |

评分时不要只看"有没有六个标题"，还要检查标题下的内容是否承担了对应功能。例如 `三、代码对比` 不能写成 API 清单，`四、实现方式` 不能写成另一个独立主题。

### 代码示例规范

借鉴《Effective Java》的代码呈现风格：

- **先反后正**：先展示常见错误 → 标注 ❌ → 给出推荐写法 → 标注 ✅ → 说明原因
- **最小可理解**：只保留与当前知识点相关的代码，删除无关样板
- **关键行注释**：在重要行旁加 `// 关键说明`，而非大段注释
- **输出标注**：代码后用 `// 输出: xxx` 或 `// 编译错误: xxx` 标明运行结果
- **示例可迁移**：变量名、类名、场景要贴近真实开发，避免 `A/B/C`、`foo/bar` 这类无语义命名
- **边界可见**：涉及异常、空值、并发、类型转换、资源关闭时，必须展示或说明失败边界
- **版本明确**：涉及 Java 8、Java 9、Java 16、Java 17+ 等版本特性时，要标注最低版本

### 主题拆分与衔接

- 一个文件只讲一个中心主题，避免把两个弱相关主题强行拼接
- 如果确实需要合并两个主题，必须在"问题引入"中说明它们的关系，并在"小结"中分别总结
- 已在前文讲过的表格或概念，后文引用即可，不要整段重复
- 后续章节可以承接前文，但必须提供最小上下文，不能要求读者回翻才能理解
- 多文件主题内部要保持递进关系：前一节解决基础问题，后一节解决扩展、边界或工程化问题

### 学习路径设计

借鉴经典书籍的知识编排逻辑：

1. **先会后懂**（Head First Java 风格）：先用起来 → 理解原理 → 掌握边界
2. **安全 > 功能 > 性能**（Java Concurrency in Practice 风格）：先保证正确性，再优化
3. **标准库优先**（Core Java 风格）：优先讲解 JDK 标准实现，第三方作为扩展
4. **问题 → 方案 → 原理**（Effective Java 风格）：不堆砌 API，按问题组织内容
