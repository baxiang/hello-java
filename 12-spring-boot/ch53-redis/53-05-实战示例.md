# 53.5 实战示例：缓存与锁（生活类比：仓库管理）

## 一、先问一个问题

**问题**：如何在一个电商系统中同时使用缓存和分布式锁？

### 生活例子

仓库管理：
- 热销商品放前台（缓存热门数据）
- 库存重地加把锁（分布式锁保护）
- 进出库要登记（缓存与数据库同步）

前台货架让顾客快速拿到商品，不需要每次跑仓库。但库存数量是核心数据，必须保证准确——同一时间只能有一个人修改库存，否则就会出现"超卖"问题。

## 二、缓存商品数据

### 2.1 查询商品（先查缓存）

```java
@Service
public class ProductService {
    
    @Autowired
    private ProductMapper productMapper;
    
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;
    
    public Product findById(Long id) {
        String key = "product:" + id;
        
        // 查缓存
        Product product = (Product) redisTemplate.opsForValue().get(key);
        if (product != null) {
            return product;
        }
        
        // 查数据库
        product = productMapper.findById(id);
        if (product != null) {
            // 写入缓存，30 分钟过期
            redisTemplate.opsForValue().set(key, product, 30, TimeUnit.MINUTES);
        }
        
        return product;
    }
}
```

**缓存模式说明**：这就是经典的 Cache-Aside 模式。先查缓存，命中则返回；未命中则查数据库并回填缓存。

### 2.2 更新商品（同时更新缓存）

```java
@Service
public class ProductService {
    
    @Autowired
    private ProductMapper productMapper;
    
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;
    
    public void update(Product product) {
        // 更新数据库
        productMapper.update(product);
        
        // 更新缓存
        String key = "product:" + product.getId();
        redisTemplate.opsForValue().set(key, product, 30, TimeUnit.MINUTES);
    }
    
    public void delete(Long id) {
        productMapper.delete(id);
        redisTemplate.delete("product:" + id);
    }
}
```

**注意**：更新时推荐"先更新数据库，再更新缓存"。删除时推荐"先删除缓存，再删除数据库"。这样可以最大程度减少数据不一致的时间窗口。

## 三、库存扣减（分布式锁）

```java
@Service
public class SeckillService {
    
    @Autowired
    private RedissonClient redissonClient;
    
    @Autowired
    private ProductMapper productMapper;
    
    public void seckill(Long productId, Long userId) {
        String lockKey = "seckill:lock:" + productId;
        RLock lock = redissonClient.getLock(lockKey);
        
        if (lock.tryLock()) {
            try {
                // 检查库存
                Product product = productMapper.findById(productId);
                if (product.getStock() <= 0) {
                    throw new RuntimeException("库存不足");
                }
                
                // 扣减库存
                productMapper.decreaseStock(productId);
                
                // 创建订单
                // ...
                
            } finally {
                lock.unlock();
            }
        } else {
            throw new RuntimeException("系统繁忙");
        }
    }
}
```

**锁的细节**：
- `tryLock()` 尝试获取锁，获取失败说明有其他线程正在处理同一个商品的库存
- `finally` 块保证锁一定会被释放，即使业务逻辑抛出异常
- 每个商品用独立的锁 key，不同商品的秒杀互不阻塞

## 四、常见疑问

**Q1：缓存和数据库数据不一致怎么办？**
A：更新时先更新数据库，再更新缓存。删除时先删除缓存，再删除数据库。

**Q2：缓存穿透（查询不存在的数据）如何解决？**
A：缓存空值或使用布隆过滤器。设置较短的过期时间。

**Q3：锁的超时时间怎么设置？**
A：Redisson 支持设置 `leaseTime`，也可以启用看门狗机制自动续期。建议根据业务执行时间合理设置，避免锁长期不释放。

**Q4：高并发下缓存失效（缓存雪崩）怎么办？**
A：给不同 key 设置随机的过期时间，避免大量缓存在同一时刻失效。同时可以加互斥锁只让一个线程去查库重建缓存。

## 五、小结

| 场景 | 方案 | 关键点 |
|------|------|--------|
| 商品查询 | 缓存 + 数据库 | 先查缓存，没有再查库 |
| 商品更新 | 双写 | 更新库 + 更新缓存 |
| 库存扣减 | 分布式锁 | 防止超卖 |
